> 研究周期：2026-06-04 ~ 2026-06-07（4 天）

## 一、解决的问题

**微信 4.1.x 的聊天消息到底存储在哪里？如何导出？**

经过 4 天的系统研究，答案如下：

**存储位置：** Weixin.exe 进程的堆内存中。
**存储格式：** 39B 紧凑结构 + 其他辅助格式。
**导出方法：** pymem 扫描前缀 `XX 02 05 09 01 01 04` → 提取内联文本。
**导出规模：** 一次扫描 ~240+ 条消息，准确率 95%。

## 二、核心发现

### 消息不在传统位置

| 可能性 | 结论 | 证据 |
|--------|------|------|
| SQLite 数据库 | ❌ 不存在 | 35,063 文件搜索 0 匹配 |
| LevelDB / MMKV | ❌ 不存消息 | 仅含 Web 配置 |
| 磁盘文件 | ❌ 不在磁盘 | 唯一消息 TEST_RAY 全盘搜索 0 匹配 |
| **Weixin.exe 内存** | **✅ 唯一位置** | **pymem 57 匹配 + Frida 双重验证** |

### 消息架构

```
用户按 PageUp
  → Caller1 (DLL 0x01683b08)
    → GetPagedMessages (DLL 0x016ff6b0)
      → FUN_1816c2a20 (步长 0x2d8 遍历元数据)
      → 39B 紧凑结构 ← 文本在此
```

### 紧凑结构格式

```
[前缀 7B: XX 02 05 09 01 01 04]
[文本内容: UTF-8，变长]
[04 分隔符]
[序号: 2B 小端 u16]
[会话标签: 8B]
[用户标识: 2B]
[头部: 7~8B]
[填充: 对齐]
```

- 前缀首个字节变化（`1b`/`25`/`23` 等），核心六字节 `02 05 09 01 01 04` 固定
- 文本内联存储——这解释了为什么所有指针追踪方法都找不到它
- 包含序号和会话标签，可用于按会话聚类和重建时间线

### 关键函数地址（v4.1.10.29）

| 函数 | DLL 偏移 | 说明 |
|------|---------|------|
| Caller1 | `0x01683b08` | 翻页入口 |
| GetPagedMessages | `0x016ff6b0` | 消息加载函数 |
| FUN_1816c2a20 | `0x016c2a20` | 消息过滤遍历（步长 0x2d8） |
| FUN_1816c2a20 循环体 | `0x016c2a76` | 遍历时 R14 指向当前元素 |
| GetMessageListBySvrIds | `0x016f3b30` | 消息列表查询 |
| FUN_185b91d80 | `0x05b91d80` | 上游消息处理 |
| Weixin.dll 基址 | `0x7fff08db0000` | ASLR 后运行时地址 |

### 与旧版 (4.1.9.56) 的差异

| 项目 | 旧版 | 新版 |

## 三、已证伪的路径

以下方向浪费了大量时间，**不要重复尝试**：

| 方向 | 原因 |
|------|------|
| 数据库解密 | V4 不使用 SQLite |
| 自动翻页（19 种方法） | Chromium WebView 拦截所有程序化输入 |
| 文件 API Hook | 翻页时不读文件 |
| 反向指针追踪文本 | 文本内联存储，无指针引用 |
| 0x2d8 结构体后处理 | 生命周期极短，离开页面即释放 |
| FUN_1816c2a20 下游函数抓文本 | 文本不在这些函数中 |
| FUN_1816f3510 缓存对象 | 0x2f0 对象不包含文本 |

## 七、结论

**微信 4.1.x 的聊天消息可以通过扫描进程内存中的紧凑结构来导出。** 该方法不需要 Hook、不需要注入、不需要管理员权限（理论上），准确率 95%。

主要限制：
1. 一次扫描只能获取当前内存中的消息（约 200-500 条）
2. 不同来源/时间的消息可能使用不同存储格式
3. 无法获取已被内存回收的历史消息

这些限制在当前版本下难以突破，因为消息存储格式不是一个统一的数据库，而是多个并行的内存缓存结构。
