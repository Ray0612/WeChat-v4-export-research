# 微信 4.1.9.56 导出项目 — 第五次迭代报告

## 项目状态

**项目阶段：** 已从"数据库解密问题"转变为"内存数据定位问题"

**第四次迭代结论：**
- 聊天记录**不在磁盘**（35,063 文件搜索，0 匹配）
- 聊天记录**在 Weixin.exe 内存**（74+ 匹配）

## 第五次迭代成果

### 1. 发现两种消息存储格式

#### 格式A：紧凑结构（消息对象数组）

在地址 `0x1f4726220ee` 附近发现的连续消息队列：

```
偏移    内容
0074    ......HELLO_5......     ← 倒序排列（最新在前）
0090    ......HELLO_4......
00b0    ......HELLO_3......
00d0    ......HELLO_2......
00fc    ......HELLO_1......
011c    ......TEST_AAAAA_11111...
```

**特征：**
- 消息间固定间距 34 字节
- 倒序排列（新消息在前面）
- 每个消息结构：
  ```
  [7字节前缀] [消息内容] [04] [序号2字节] [???] [地址] [常量] [偏移] [结尾]
  ```

**序号验证：**
| 消息 | 序号（小端序） | 数值 |
|------|------|------|
| HELLO_5 | 4f 01 | 335 |
| HELLO_4 | 4e 01 | 334 |
| HELLO_3 | 4d 01 | 333 |
| HELLO_2 | 4c 01 | 332 |
| HELLO_1 | 4b 01 | 331 |
| TEST_AAAAA | 4a 01 | 330 |

**常量 `6a 22 5e`** 出现在所有紧凑消息中，可能是当前登录用户标识

#### 格式B：ProtoBuf 格式（完整消息对象）

在地址 `0x1f40b1111d4` 发现的完整消息结构：

```
HELLO_1 ProtoBuf:
┌─ 08 01              → field 1 = 1（消息类型？）
├─ 12 59              → field 2 = sub-message（89字节）
│    ├─ 0a 0c         → field 1 = "filehelper"（接收者）
│    └─ 12 07         → field 2 = "HELLO_1"（内容）
├─ 18 01              → field 3 = 1
├─ 20 c9 bc 89 d1 06  → field 4 = 27145628745（MsgID）
├─ 28 cc e7 89 f8 09  → field 5 = 40645169484（MsgID/时间戳）
└─ 32 32              → field 6 = <msgsource>...（XML元数据）
```

### 2. 发现结构化记录区（Key-Value 存储）

在 `0x1f448dd44d0` 发现的结构化记录区，包含多个类型的字段：

| 偏移 | 内容 | 类型码 | 说明 |
|------|------|--------|------|
| 0x100 | TEST_OTHER_CHAT | - | 消息内容 |
| 0x110 | _938274615 | - | TEST_RAY 尾缀 |
| 0x160 | wxid_049vxvhc4asy22 | 0x73 | **对方微信号** |
| 0x1f0 | 47646221656@chatroom | 0x76 | **群聊ID** |
| 0x220 | 科技1班班委群 | 0x77 | **群名（中文UTF-8）** |

每条记录结构：
```
[?? 2B] [30] [??] [9f 00] [类型码] [02] [90] [字符串数据] [00 填充]
```

类型码连续：0x73→0x74→0x75→0x76→0x77→0x78

### 3. 消息存储对比实验结论

| 实验 | 结果 |
|------|------|
| 同联系人连续发5条消息 | 紧凑结构中找到全部5条（倒序、34字节间距） |
| 切换联系人发消息 | 接收者 ID（wxid）不同 |
| 发送至群聊 | 群ID（@chatroom）+ 群名（中文）出现在记录区 |
| 关闭/重开窗口 | 消息仍在内存（至少短期内缓存） |

## 完整证据链

```
用户发消息
    ↓
消息进入 Weixin.exe 堆内存
    ↓
以两种格式存储：
    ├── 紧凑格式（消息对象数组，34字节间距）
    └── ProtoBuf格式（完整消息，含接收者+MsgID+源信息）
    ↓
接收方信息存储在结构化记录区
    ↓
磁盘上不保留任何聊天记录
```

## 已确认字段映射

| 字段 | 发现位置 | 证据 |
|------|----------|------|
| 消息内容 | 两种格式都有 | HELLO_1 / TEST_OTHER_CHAT |
| 序号/Seq | 紧凑格式 | 335→330 递减 |
| 接收者wxid | 记录区0x73 | wxid_049vxvhc4asy22 |
| 群聊ID | 记录区0x76 | 47646221656@chatroom |
| 群名 | 记录区0x77 | 科技1班班委群 |
| 会话常量 | 紧凑格式内 | 6a 22 5e（所有消息共有） |

## 后续可行性

理论上可以通过内存扫描实现"运行时导出聊天记录"：
1. 定位消息内容字段（已在两种格式中找到）
2. 定位对应的接收者/群聊信息（在结构化记录区中，通过类型码索引）
3. 遍历消息队列提取所有内容
4. 关联接收者信息

但这需要开发一个专门的内存读取工具，复杂度和工作量都不小。
