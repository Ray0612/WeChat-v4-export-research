# M9 — Storage Path Discovery V1

> 实验时间：2026-06-06
> 方法：Frida Hook NtCreateFile/NtReadFile/CreateFileW/CreateFileMappingW/MapViewOfFile
> 监控范围：翻页前后 90 秒内的所有文件操作

---

## 核心发现

### 翻页期间无消息数据文件访问

Hook 了 6 个文件 API 入口，翻页期间捕获的**唯一文件访问**：

| 文件 | 打开次数 | 作用 |
|------|---------|------|
| `xwechat/log/upload_size_record.ini` | 6 | 日志上传统计（与消息无关）|
| `kvcomm/` 统计文件 | 2×10 | KV 存储通信统计（与消息无关）|

**没有发现任何 .db / .dat / .bin / .ldb 文件的读取。**

### CreateFileMappingW / MapViewOfFile 未触发

翻页期间没有触发任何内存映射文件操作。

### NtReadFile 总计数：10 次

且全部指向 `kvcomm` 统计文件的读取，与消息内容无关。

---

## 结论

**GetPagedMessages 在翻页时不从磁盘读取数据。**

证据：
1. 翻页期间的 143 次 GetPagedMessages 调用 → 0 次相关的文件操作
2. 6 个不同层次的 API Hook（ntdll → kernelbase → kernel32）全部确认无读取
3. 内存映射文件 API 同样未触发
4. 唯一的文件操作是后台统计数据刷写（kvcomm）

### 数据加载时机

消息数据必然在更早的时间点已加载到内存中：

```
微信启动 → 用户登录 → 进入聊天窗口 → [数据从磁盘加载到内存]
                                                  ↓
                                              内存中
                                                  ↓
        用户翻页 → GetPagedMessages → [从内存缓存读取，无文件 I/O]
```

这意味着需要：
- **在微信启动时就开始 Hook** 才能捕获初始文件读取
- 或在进程启动后枚举已打开的文件句柄/内存映射

### 对存储搜索的影响

| 之前以为 | 实际 |
|---------|------|
| 翻页时读文件 → 容易定位 | 翻页时不读文件 → 已全部在内存 |
| 翻页时触发的 I/O 可追踪 | 数据在启动时/首次打开聊天时加载 |
| 用 Hook 文件 API 可找到存储 | 需要 Hook 启动阶段或另寻方法 |
